Monolith

  • There is a MainPage with internal routing.
  • We add AboutPage - we have to extend components, storage, libraries.

Advantages of a monolith:

  • Ease of implementation
  • E2E test support
  • Easy deployment
  • Single stack/framework

Minuses of a monolith:

  • it becomes so much stuff that it's unclear how to work with it all
  • when we change one thing in the architecture, we need to change other parts
  • scaling can be problematic
  • all the parts are highly dependent on each other

What are Microfrontends?

Microfrontends are an architectural pattern where the frontend is broken down into independent modules that:

  • Are developed by separate teams
  • Can be deployed independently
  • Can be written in different technologies (React, Vue, Angular)

How They Differ from Microservices

  • One DOM.
  • One Local Storage
  • One URL.

Why Use Microfrontends?

1. Everything is connected. If something breaks, it takes longer to figure out

2. One team doesn't have time to do everything

3. Technology lock-in

Microfrontend Benefits

✅ Development flexibility - teams choose their own tech

✅ Independent deployment - update one module without breaking others.

✅ Scalability - easy to add new features

✅ Gradual migration - rewrite legacy code piece by piece

Logic

How it works?

1. Client-Side Composition

  • Modules are loaded into the browser and assembled in the runtime
  • Tools: single-spa, Module Federation (Webpack)
  • Microfrontends combine into single app in the browser

2. Server-Side Composition

  • Server assembles HTML from different microfrontends
  • Example: Podium
  • Better initial load performance and SEO

Challenges and Solutions

1. Duplicate dependencies and increasing the size of the bundles

Challenge: Each microfrontend may include its own library versions, increasing total size.

Solutions:

  • Use Module Federation for shared dependencies

2. Integration and isolation difficulties

Challenge: Different micro frontends can conflict over global styles, variables or events.

Solutions:

  • Use CSS modules or CSS-in-JS for style scoping
  • Standardize naming conventions

3. Testing and Debugging Complexity

Challenge: Testing and debugging become more complex due to distributed architecture.

Solutions:

  • Implement unit and integration tests per microfrontend

4. DevOps and CI/CD Complexity

Challenge: And managing the deployment and upgrades of multiple microfrontends requires additional effort.

When NOT to Use Microfrontends

Not a Silver Bullet!

🚫 Small projects (overhead not justified)

🚫 Single team (monolith may suffice)

🚫 Strict performance requirements (extra module loading)

💡 Alternative: Modular monolithic frontend
